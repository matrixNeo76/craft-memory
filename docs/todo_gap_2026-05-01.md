### FASE 1: Critical Context Preservation (Event-based)

**Obiettivo**: Preservare contesto critico prima che venga perso

---

#### 1.1 PreCompact → Emergency Memory Save

**Event**: `PreCompact`
**Trigger**: Prima che il contesto venga compattato dal framework
**Perché**: Quando il contesto si riempie, Claude "dimentica" tutto ciò che non è in memoria. Questa automazione salva automaticamente le informazioni critiche PRIMA della compattazione.

```json
{
  "name": "Memory: PreCompact Emergency Save",
  "permissionMode": "allow-all",
  "actions": [
    {
      "type": "prompt",
      "prompt": "PRE-COMPACT EMERGENCY PROTOCOL:\n\nIl contesto sta per essere compattato. Salva IMMEDIATAMENTE:\n\n1. Call get_recent_memory(scope='workspace', limit=5) per vedere cosa è già salvato\n2. Review la conversazione corrente per:\n   - Decisioni prese (call remember category='decision' importance=9)\n   - Facts scoperti (call upsert_fact)\n   - Open loops aperti (call add_open_loop se necessario)\n3. Create un breve snapshot con remember(category='note' importance=7) che riassume:\n   - Cosa stavi facendo\n   - Dove sei arrivato\n   - Cosa manca da fare\n\nBe concise but thorough. Questo garantisce continuità dopo la compattazione."
    }
  ]
}
```

#### 1.2 SessionStatusChange → Done → Quality Check + Archive

**Event**: `SessionStatusChange` con `matcher: "^done$"`
**Trigger**: Quando l'utente marca una sessione come "done"
**Perché**: Una sessione completata è un momento ideale per valutare qualità, consolidare apprendimenti, e preparare per archiviazione.

```json
{
  "name": "Memory: Session Done - Quality & Archive",
  "matcher": "^done$",
  "permissionMode": "allow-all",
  "actions": [
    {
      "type": "prompt",
      "prompt": "SESSION COMPLETED PROTOCOL:\n\nLa sessione è stata marcata come 'done'. Esegui:\n\n1. **Quality Assessment**:\n   - Valuta la sessione: è stata produttiva? (0.0-1.0)\n   - Se score >= 0.7, call rate_session(score, notes)\n\n2. **Knowledge Consolidation**:\n   - Review memories create in questa sessione (search_memory by date)\n   - Se trovi facts stabili, call upsert_fact\n   - Se trovi procedure ripetibili, call save_procedure\n\n3. **Loop Cleanup**:\n   - Call list_open_loops()\n   - Se ci sono loop risolti, call close_open_loop\n\n4. **Handoff Summary**:\n   - Call generate_handoff() per documentare per la prossima sessione\n\nPresenta un breve report all'utente."
    }
  ]
}
```

### FASE 2: Security & Compliance Events (Event-based)

**Obiettivo**: Tracciare eventi security-relevant e garantire audit trail

---

#### 2.1 PermissionModeChange → Audit Log + Memory

**Event**: `PermissionModeChange`
**Trigger**: Quando cambia il permission mode (safe → ask → allow-all)
**Perché**: Escalation di permessi è un evento security-critical. Deve essere tracciato per audit e debugging.

```json
{
  "name": "Memory: Permission Escalation Audit",
  "conditions": [
    {
      "condition": "state",
      "field": "permissionMode",
      "from": "safe",
      "to": "allow-all"
    }
  ],
  "actions": [
    {
      "type": "prompt",
      "prompt": "SECURITY AUDIT - Permission Escalation:\n\nPermission mode escalated from 'safe' to 'allow-all'.\n\n1. Call remember(\n   content='Permission escalated from safe to allow-all in session ${CRAFT_SESSION_NAME}',\n   category='change',\n   importance=8,\n   tags=['security', 'audit', 'permission-escalation']\n)\n\n2. Log a WARNING to the user explaining what this means:\n   - Auto-approval attivo per tutte le operazioni\n   - Fare attenzione a operazioni irreversibili\n\nThis creates an audit trail for security review."
    }
  ]
}
```

### FASE 3: Proactive/Agentic Behaviors (Label-driven Intelligence)

**Obiettivo**: Automazioni che triggerano comportamenti intelligenti, non solo storage

---

#### 3.1 LabelAdd `procedure-candidate` → Extract Procedure

**Event**: `LabelAdd` con `matcher: "^procedure-candidate$"`
**Trigger**: Quando l'utente o l'agent aggiunge label `procedure-candidate`
**Perché**: Spesso si ripetono pattern di lavoro. Questa automazione estrae automaticamente una procedure riutilizzabile dalla sessione.

```json
{
  "name": "Memory: Extract Procedure from Session",
  "matcher": "^procedure-candidate$",
  "permissionMode": "allow-all",
  "actions": [
    {
      "type": "prompt",
      "prompt": "PROCEDURE EXTRACTION PROTOCOL:\n\nLabel 'procedure-candidate' added. This session contains a repeatable workflow.\n\n1. **Analyze Session**:\n   - Review la conversazione per identificare il pattern ripetibile\n   - Identifica: trigger context, steps, expected outcome\n\n2. **Extract Procedure**:\n   Call save_procedure(\n   name='descriptive-kebab-case-name',\n   trigger_context='When to apply this procedure',\n   steps_md='# Step-by-step markdown',\n   confidence=0.7\n   )\n\n3. **Link to Source**:\n   - Se ci sono memories rilevanti, nota i loro ID per source_memory_ids\n\n4. **Report**:\n   Presenta la procedure estratta all'utente per conferma/refinement.\n\nThe procedure will be retrievable via get_applicable_procedures() in future sessions."
    }
  ]
}
```

```json
{
  "name": "Memory: Suggest Graph Links",
  "matcher": "^link-suggestions$",
  "permissionMode": "allow-all",
  "actions": [
    {
      "type": "prompt",
      "prompt": "GRAPH LINK SUGGESTION PROTOCOL:\n\n1. **Get Recent Context**:\n   Call get_recent_memory(limit=5) per ottenere memories recenti\n\n2. **For each recent memory**:\n   Call find_similar(memory_id, top_n=3, auto_link=False)\n\n3. **Analyze Results**:\n   - Identifica coppie con similarity score alto (>0.7)\n   - Determina il tipo di relazione: extends, caused_by, semantically_similar_to\n\n4. **Present Suggestions**:\n   Mostra all'utente:\n   - Memory A ↔ Memory B (score: 0.85) → Suggested: extends\n   - Memory C ↔ Memory D (score: 0.72) → Suggested: semantically_similar_to\n\n5. **Optional Auto-link**:\n   Se l'utente approva, call link_memories(source_id, target_id, relation)\n\nThis helps build a richer knowledge graph without manual effort."
    }
  ]
}
```

### FASE 4: Scheduled Intelligence (Cron-based)

**Obiettivo**: Automazioni schedulate che vanno oltre il daily maintenance

---

#### 4.1 Weekly Memory Review

**Event**: `SchedulerTick` con `cron: "0 10 * * 1"` (Lunedì alle 10:00)
**Trigger**: Ogni lunedì mattina
**Perché**: Il daily maintenance è conservative. Il weekly review è più approfondito e genera report per l'utente.

```json
{
  "name": "Memory: Weekly Review Report",
  "cron": "0 10 * * 1",
  "timezone": "Europe/Rome",
  "permissionMode": "allow-all",
  "labels": ["scheduled", "weekly-review"],
  "actions": [
    {
      "type": "prompt",
      "prompt": "WEEKLY MEMORY REVIEW:\n\n1. **Stats Overview**:\n   Call memory_stats() per overview\n\n2. **Analyze Week**:\n   - Call memory_diff(since_epoch=7 days ago)\n   - Identify: new decisions, new facts, open loops opened/closed\n\n3. **Procedure Health**:\n   - Call list_procedures(status='active')\n   - Check get_procedure_outcomes per each\n   - Identify procedures with low success_rate\n\n4. **Quality Report**:\n   - Call get_high_quality_sessions(min_score=0.7, limit=5)\n   - Summarize what made these sessions successful\n\n5. **Recommendations**:\n   - Suggest procedures to deprecate (low success)\n   - Suggest memories to promote to core\n   - Suggest stale loops to close\n\nPresent a comprehensive weekly report to the user."
    }
  ]
}
```
